# 電子書 01 大綱

## 書名
**《AI Skills / Plugins 上線前安全檢查清單》**

## 副標
給開發者、AI 團隊與技術負責人的實戰指南：用最小成本建立 skill / plugin 上線前的審查、限權與稽核流程。

## 這本書要解決的問題
多數團隊不是不知道 AI skills / plugins 很方便，而是不知道該怎麼在不拖慢速度的前提下，把它們安全地接進 production。

這本書不是講空泛的 AI 安全理論，而是要提供一套 **可直接照抄的上線前安全檢查框架**，幫團隊回答三個問題：

1. 這個 skill / plugin 到底能不能裝？
2. 如果要裝，應該限制到什麼程度？
3. 上線後，怎麼知道它沒有開始亂搞？

---

# 建議目錄

## 前言：不是不能裝，是不能亂裝
### 本章重點
- 為什麼 AI skills / plugins 正在變成新的供應鏈入口
- 大多數團隊真正卡住的不是技術，而是信任
- 本書承諾：不用大公司預算，也能建立最小可行防線

### 可先寫的內容方向
- 開場先講現實：團隊很想用 agent，但不敢放進真實工作流
- 點出問題不在功能，而在於誰能被信任、誰該被限制、出事時誰看得見
- 明確表態：本書講的是可執行流程，不是理論總整理

---

## 第 1 章：為什麼 AI Skill / Plugin 是新的供應鏈風險
### 本章重點
- skill / plugin 與傳統套件、SaaS 整合、腳本自動化的差異
- 為什麼它們的風險更接近「可執行的外部能力」
- 為什麼 agent 生態讓導入速度變快，但風險審查通常沒跟上

### 要講的核心觀念
- 這不只是裝一個功能，而是引入一個會讀、會寫、會呼叫外部資源的行為單元
- 一旦 skill 有權限，影響的是資料、流程、決策與外部系統
- 問題不是 skill 會不會出錯，而是出錯時你有沒有控制權

### 可插入的例子
- 看似只是抓資料的 skill，實際上能外連、寫檔、碰 token
- 一個第三方 plugin 讓 agent 取得比原本預期更大的執行權限

---

## 第 2 章：最常見的 7 種風險
### 本章重點
把抽象風險拆成可辨識、可檢查的類型。

### 建議風險分類
1. **外連風險**：未預期的 API、webhook、資料傳送
2. **Secrets 存取風險**：讀取環境變數、憑證、token、敏感檔案
3. **動態執行風險**：eval、shell、遠端下載後執行
4. **過大權限風險**：能讀太多、寫太多、跑太多
5. **依賴污染風險**：轉帶不可信套件、版本飄移、隱性供應鏈問題
6. **資料外洩風險**：把內部資料送到不該去的地方
7. **不可稽核風險**：沒有 log、沒有 trace、沒有 redaction、沒有審計線索

### 每個風險建議統一段落結構
- 風險長什麼樣
- 常見警訊
- 為什麼危險
- 最低限度的控制方式

---

## 第 3 章：安裝前該檢查什麼
### 本章重點
建立安裝前的 intake / vetting 概念，先把垃圾擋在門外。

### 應涵蓋的檢查項
- 來源是否可信
- 發布者是否可辨識
- 授權是否合理
- 依賴是否異常
- 是否有可疑外連、動態下載、動態執行
- 是否要求超出用途的權限
- 是否有文件說明資料流向與執行行為

### 可轉成 checklist 的問題
- 誰發布的？
- 為什麼需要這些權限？
- 會碰哪些資料？
- 有沒有任何動態執行或遠端程式碼載入？
- 如果今天出事，我們看得到 log 嗎？

---

## 第 4 章：執行前該限制什麼
### 本章重點
不是通過審查就放飛，而是進 production 前必須限權。

### 應涵蓋的控制措施
- Least privilege
- 沙箱隔離
- 檔案系統存取限制
- 網路 egress allowlist
- secrets 最小暴露
- 明確的人類審批點
- 對高風險操作加入 confirmation / policy gate

### 可講的原則
- 通過 vetting ≠ 可以拿完整權限
- 功能需求與系統權限要拆開看
- 權限應以任務為單位，而不是以 skill 為單位無限放大

---

## 第 5 章：執行中該監控什麼
### 本章重點
如果沒有運行期觀測，你根本不知道 skill 何時開始偏掉。

### 應涵蓋的監控面向
- 外連目標
- 敏感檔案讀取
- secrets 使用行為
- 指令執行情況
- 權限異常升高
- 行為模式偏移（drift）
- 錯誤率與異常重試

### 可講的重點
- log 不是越多越好，而是要能回答問題
- trace 要可追溯，但敏感資訊要 redaction
- 觀測性不是 debug 功能，而是安全控制的一部分

---

## 第 6 章：一份可以直接用的安全審查 Checklist
### 本章重點
把前面內容濃縮成可落地的表單。

### 建議 Checklist 分區
1. 基本資訊
   - skill 名稱
   - 來源
   - 發布者
   - 版本
   - 授權
2. 靜態風險檢查
   - 可疑外連
   - 可疑指令執行
   - 可疑檔案存取
   - 依賴風險
3. 執行權限評估
   - 需要哪些權限
   - 哪些權限可以拿掉
   - 是否需要人類審批
4. 上線條件
   - 是否需 sandbox
   - 是否需額外監控
   - 是否允許進 production
5. 稽核資訊
   - 審查者
   - 日期
   - 風險等級
   - 決策理由

### 產出形式建議
- Markdown 版
- 表格版
- Notion / Google Docs 可複製版

---

## 第 7 章：小團隊可直接採用的最小 SOP
### 本章重點
不是每個團隊都要 enterprise 流程，但每個團隊都需要最低限度的治理。

### 建議 SOP 流程
1. 提出 skill 導入需求
2. 安裝前 intake
3. L1 基礎靜態檢查
4. L2 沙箱測試
5. 權限裁切
6. 決策：允許 / 限制 / 暫緩 / 拒絕
7. 上線後觀測與定期複審

### 可補的重點
- 一人團隊怎麼做簡化版
- 三到十人團隊怎麼分工
- 什麼情況需要升級成正式安全審批

---

## 第 8 章：如何把這件事產品化 —— SkillGate 的角色
### 本章重點
把手動流程自然導向產品，而不是硬廣。

### 建議切法
- 手動做法能跑，但會痛在哪
- 團隊一多、skill 一多，流程會在哪裡失控
- SkillGate 如何處理：
  - intake
  - static vetting
  - sandbox runtime analysis
  - policy approval gate
  - trusted registry
  - runtime admission control
  - observability / audit

### 應避免的寫法
- 不要像 sales brochure
- 要像：當手動流程長到某個規模，產品自然變成合理答案

---

## 結語：先建立最低限度的信任層，再擴大自動化
### 本章重點
- 安全不是讓團隊變慢，而是讓團隊敢把 agent 放進真正有價值的流程
- 可被信任的 skill 流程，是 AI automation 能進 production 的前提
- 先把最小防線做好，再談規模化

# 前言草稿（v0）
很多團隊不是不想用 AI skills 或 plugins，而是不敢把它們接進真的工作流。

原因很現實：你不知道它會碰到哪些資料、不知道它到底需要哪些權限，也不知道它上線後會不會開始做出超出預期的事情。

這種不確定性，會直接卡住導入。結果通常不是「安全做對了」，而是兩種極端：不是完全不敢用，就是一股腦裝上去，等出事再補洞。

這本書的目的，就是把這件事拉回務實。

我們不談空泛的 AI 安全口號，也不預設你有大公司那種完整安全團隊。相反地，這本書要提供的是一套小團隊也能照抄的做法：怎麼在 skill / plugin 上線前先做審查、上線時先做限權、上線後留下可稽核的軌跡。

你不需要一開始就做到完美，但你至少要先有一道門。

而這本書，就是幫你把那道門搭起來。
