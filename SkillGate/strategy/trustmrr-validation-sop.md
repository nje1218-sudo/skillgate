# SkillGate / 電子書 / 模板包：TrustMRR 驗證 SOP

## 目的
這份 SOP 用來規範：

- 我們怎麼用 **TrustMRR** 當成產品驗證來源之一
- 它可以幫我們驗證什麼
- 它不能替代什麼
- 怎麼把它實際用在 **SkillGate、電子書、模板包** 的商業判斷上

---

## 一、核心原則

### 原則 1：TrustMRR 是驗證來源之一，不是唯一真理
TrustMRR 可以幫我們判斷：
- 市場上有沒有真實付費
- 哪些類型的產品真的有人買
- 哪些題材不只是社群吵，而是有收入證據

但它**不能直接證明**：
- 我們的切角一定會贏
- 我們的文案一定會轉換
- 我們的電子書一定有人買

---

### 原則 2：不要單押單一來源
每次做產品驗證，都至少要看 4 種來源：

1. **社群熱度**
   - OpenClaw 社群
   - OpenClaw 技能討論區
   - GitHub
   - Reddit
   - X（Twitter）討論度

2. **產品存在**
   - 市場上有沒有相近商品、工具、服務、電子書

3. **付費證據**
   - TrustMRR
   - 其他收入/交易線索平台

4. **我們自己的驗證**
   - landing page
   - waitlist
   - preorder
   - 小額付費測試

---

### 原則 3：TrustMRR 適合驗證大方向，不適合取代微觀測試
TrustMRR 最適合回答的是：
- 這個領域有沒有真 MRR
- 類似需求是否值得做
- 這類產品通常怎麼包裝

TrustMRR 不適合直接回答的是：
- 這個書名一定會賣嗎
- 這個 CTA 會不會轉
- 這個價位使用者會不會買

這種仍要靠自己的頁面與小額測試。

---

## 二、TrustMRR 驗證流程

## Step 1｜先定義我們要驗證什麼
每次查 TrustMRR 前，先寫清楚要驗證的是哪一類問題：

### A. 市場存在性
例：
- AI automation 是否有人買單？
- OpenClaw deployment / agent setup 是否有商業價值？

### B. 商業敘事
例：
- 「更安全上線」這個敘事有沒有錢味？
- 「讓 agent 被放心拿去賺錢」這個說法有沒有對齊市場？

### C. Offer 包裝
例：
- 市場上賣得動的是工具、服務、資訊產品，還是 bundle？
- recurring revenue 模型是不是比較合理？

### D. ICP（理想客群）
例：
- 是小團隊、agency、solo builder 還是 SaaS founder 比較會付錢？

---

## Step 2｜去 TrustMRR 找相近品類
優先觀察以下類型：
- AI automation
- AI agent / chatbot / assistant
- SEO / content automation
- sales / outreach / lead gen
- operations / workflow / productivity
- OpenClaw deployment / setup / infra
- security / compliance / risk-reduction adjacent tools

### 記錄欄位
每次至少記下：
- 產品名稱
- 類型（工具 / 服務 / marketplace / agency / template / education）
- 核心承諾
- 目標客群
- 大概的商業模式
- 是否有 verified revenue
- 和 SkillGate 的相關度（高 / 中 / 低）

---

## Step 3｜不要只看收入，還要看它到底在賣什麼
TrustMRR 的價值不只是 MRR 數字，而是：

### 看這些訊號
1. **它賣的結果是什麼**
   - 省時間
   - 提升收入
   - 自動化
   - 降成本
   - 降風險

2. **它賣的是什麼型態**
   - SaaS
   - 顧問服務
   - agency service
   - setup / deployment service
   - membership / info product

3. **它怎麼描述痛點**
   - 更快
   - 更穩
   - 更賺
   - 更省事
   - 更放心

### 對 SkillGate 特別重要的判斷點
- 市場是不是願意為「降低風險 + 維持營收」買單
- 使用者對「安全」本身不一定會付錢，但對「因為安全，系統能持續賺錢」更可能付錢

---

## Step 4｜把 TrustMRR 結果翻譯成我們自己的語言
TrustMRR 不會直接長出 SkillGate 的文案，所以要做一次翻譯：

### 原始觀察
例：
- 市場上 AI automation / outreach / deployment 有 verified revenue

### 翻譯成 SkillGate 的商業敘事
- SkillGate 不是賣安全感，而是賣：
  - 讓 AI 系統更安全上線
  - 讓 agent 能被放心用來賺錢
  - 讓自動化不因技能/插件風險而翻車

### 最終定位句
- 安全不是終點；安全是讓自動化收入能成立的前提。

---

## 三、怎麼用在不同產品上

## A. 用在 SkillGate 主產品
### 適合驗證的問題
- 這個市場願不願意為風險降低與系統可靠性買單？
- builder / 小團隊 / agency 是否願意買「更放心上線」的能力？

### 要看什麼
- TrustMRR 上是否有：
  - deployment/setup 類產品
  - AI automation agency / tool
  - 以穩定性/收入為賣點的工具

### 結論怎麼寫
不要寫：
- 市場上有人賣安全工具，所以 SkillGate 會賣

要寫：
- 市場上已證明，使用者願意為能提升營收、降低運營風險、提高自動化可靠性的產品付費。SkillGate 應定位為讓 AI 系統更安全上線、並能放心拿去賺錢的底層保障。

---

## B. 用在電子書
### 適合驗證的問題
- 這個方向是不是有商業價值，不只是知識分享？
- 市場對安全上線、風險治理、可執行 SOP 是否存在真需求？

### 不適合驗證的問題
- 這本 US$19 電子書會不會有人買
- 哪個書名最好
- 哪個副標轉換最高

### 正確用法
TrustMRR 用來支撐這句話：
- 這個大方向有真商業價值，所以電子書值得拿來當前端驗證產品

後面真正驗證電子書的，還是要靠：
- landing page
- preorder
- waitlist
- email capture

---

## C. 用在模板包
### 適合驗證的問題
- 市場是否偏好可直接拿去用的資產（而不只是內容）？
- builder 是否願意為 SOP / checklist / templates 這類縮短落地時間的東西付錢？

### 觀察角度
- 市場上有沒有很多產品在賣：
  - playbook
  - templates
  - agency assets
  - SOP bundles

### 對 SkillGate 的啟發
如果市場已經願意為「省時間、縮短試錯、直接上手」付錢，那模板包就不是附屬品，而是獨立可賣資產。

---

## 四、每次驗證要輸出的結論格式
每次用 TrustMRR 驗證完，都要產出這 5 行：

### 1. 驗證問題
這次想驗證什麼？

### 2. TrustMRR 觀察
看到哪些相近產品 / 收入訊號？

### 3. 可採信結論
TrustMRR 真正支持我們下什麼判斷？

### 4. 不能下的結論
哪些事情不能因為看到 TrustMRR 就過度延伸？

### 5. 下一步測試
要用 landing page / preorder / waitlist 驗證什麼？

---

## 五、紅線與誤用提醒

### 不要這樣用
- 因為 TrustMRR 有 AI automation 賺錢，所以 SkillGate 一定會賣
- 因為某產品有高 MRR，所以我們做電子書也能賣
- 只看收入數字，不看 offer / 客群 / 敘事 /商業模式

### 要這樣用
- 用 TrustMRR 驗證大方向
- 用社群與競品補上下文
- 用自己的頁面與小額測試驗證最後一哩

---

## 六、SkillGate 的固定使用規則
之後只要評估 SkillGate 相關產品（含電子書、模板包、Guardian、審查服務），都要至少檢查：

- OpenClaw 社群
- OpenClaw 技能討論區
- X（Twitter）討論度
- Reddit 討論
- TrustMRR
- 類似商品 / 類似服務
- 自己的 landing page / waitlist / preorder 測試

---

## 七、一句話版
TrustMRR 用來回答：**這條路有沒有人真的在付錢。**
但最後還是要靠我們自己驗證：**他們會不會為我們這個切角付錢。**
